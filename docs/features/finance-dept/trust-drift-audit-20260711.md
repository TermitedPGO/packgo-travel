## 結論一句話

這 −$10,442 不是「錢憑空不見」,也不是退款或重複記帳;它是一個 §17550 混同 + 提前掃款的問題:三筆客人訂金($15,422)被系統記成「還押在信託裡未認列」,但這個帳戶實際是當「落地即掃往 Operating #2174」的過水帳在用,訂金存進來幾天內就被掃走大半。錢都追得到(在 #2174,或還留在本帳餘額),沒有一分蒸發;真正的風險是合規面,不是現金面。

## 第二節:根因 —— 這個帳戶是「過水帳」不是「隔離信託」

信託帳 #5442 全期(2026-01-23 至 2026-06-27)共 40 筆交易,進帳合計 −$63,971.20、出帳合計 $63,297.20、淨 −$674。看整條流水,型態非常一致:

- 客人手機存款進來 → 1 至 4 天內一筆(或幾筆)等額「Online Banking transfer to CHK 2174」把錢掃往 Operating #2174。全期十幾筆客人存款,幾乎每一筆都能在事後找到金額吻合的掃款。
- Jeff 也反向從 Operating 灌錢進來當週轉:2026-01-23 一筆 −$21,000(隨後 02/02 +$7,000、03/04 +$14,000 掃回,合計剛好 21,000,週轉float 已還清)、04/27 −$4,980、05/28 −$1,120、06/01 −$504。
- 全帳戶 bankTransactionLinks = 0 筆:這 40 筆沒有任何一筆在對帳引擎裡被 claim/link 過,遞延列與掃款彼此從未被系統勾稽。

也就是說,這個帳戶被當成收款落地 + 掃往營運的過水帳在跑,客人的錢和 Jeff 的營運週轉金在裡面混同。信託該有的「收了就押著、出發才動」在這帳上並不成立。這是整條 drift 的根,也是後面合規段的核心。

補一句:帳戶標籤是「Living Trust Account」(個人生前信託),不是 CST §17550 那種客戶專用信託帳。客人訂金拍進 Jeff 的生前信託帳、再掃進營運帳,本身就是帳戶用途錯配。

## 第四節:三筆加總對得上 −$10,442 嗎

對得上,在下述模型假設下分毫不差:

```
drift = 帳戶餘額 − 應押金額 = 4,980 − 15,422 = −10,442
拆解:
  提前掃往 Operating(第5桶) = 8,908(#1鐵證) + 2,516(#2) + 1,000(#3) = 12,424
  仍留在信託帳的客人殘額(第2桶) = 15,422 − 12,424 = 2,998
  營運週轉金墊高餘額的部分         = 4,980 − 2,998 = 1,982
  −(12,424) + 1,982 = −10,442  ✓
```

殘差 / 不確定區間(誠實標明):因帳戶混同、錢可替代(fungible),「$4,980 餘額裡有多少是客人的、多少是 Jeff 灌進來的週轉金」無法從銀行資料唯一決定。兩個都站得住的讀法:

- 溫和讀法(上表):客人殘額 $2,998 仍在信託,掃出 $12,424,週轉金墊 $1,982。
- 嚴格讀法:4/27 那筆從 Operating 灌入的 −$4,980 恰好等於今天餘額 $4,980;若照這條線,今天帳上這 $4,980 其實是營運週轉金,三筆客人訂金 $15,422 已全數掃出,信託帳客人錢實質為 $0。

不論哪條,底線一致:三筆客人訂金裡,至少 $8,908(鐵證)、最多 $15,422 已在出發/認列前離開信託帳、進了 Operating。差在 −$10,442 到 −$15,422 之間,取決於你認為餘額裡的客人成分有多少。無「對外不明流出」殘差,第 6 桶為 $0。

## 第六節:CST §17550 合規評估(非法律意見)

以下是查帳觀察,不是法律意見;涉及 CST 註冊、信託帳義務與豁免請諮詢加州執業律師或 CST 合規顧問。

有合規疑慮,而且是實質的,不是形式的。加州賣旅遊者法(Sellers of Travel,Bus. & Prof. Code §17550 起)的核心要求是:收到的客戶款要放進信託帳、押到旅遊服務實際提供(或退款/沒收)才動用,除非該賣家以 CST 保證金/其他法定豁免替代信託。本案觀察到的型態與這條相衝:

1. 混同:客人訂金與 Jeff 的營運週轉金在同一帳戶進出(Operating 反覆灌入、客人款反覆掃出),不是隔離保管。
2. 提前動用:訂金在行程出發前幾天內就被掃往 Operating(#1 鐵證,#2/#3 部分),等於服務尚未提供就先把客人的錢挪去營運。
3. 帳戶錯配:承接客人款的是個人「生前信託帳」,不是客戶專用信託帳。

緩解與待確認:若這三筆對應的行程其實已經出發,那些款項已屬賺得收入,掃往 Operating 事後站得住,問題降級為「帳務認列沒跟上」而非「動了不該動的客戶錢」。關鍵事實(行程有沒有出發)DB 查不到,因為三筆 bookingId 皆 NULL、expectedRecognitionDate 皆空;必須由 Jeff 指認訂單補上。另外,若 PACK&GO 是以 CST 保證金替代信託帳的模式營運,信託義務的樣態也不同,需以實際 CST 登記狀態為準。

風險量級:就這三筆而言,金額 $15,422、其中至少 $8,908 可證已提前離開信託。錢沒有遺失(全在 Operating 或本帳),所以這比較像「保管方式與時點違規 + 帳務未認列」,不是挪用侵占。優先動作是第五節那條:先指認三筆對應訂單與出發日,能認列的認列、未出發的把等額客人款補回信託押到出發,並停止把客人款掃進混同帳戶。
