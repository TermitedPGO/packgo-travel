### webhook auditLog 豁免申報(回爐 #4)

`deferStripeBookingIncome` 與 `reverseDeferral`(trustDeferralService.ts)兩處錢寫入
**確實沒有呼叫 `_core/auditLog.ts` 的 `audit()`**。監工給的兩個選項擇一,選「T6 申報豁免」,理由:

- `audit()` 硬性要求 `ctx.user`(admin 已登入 context),沒有就 `log.warn` 靜默 return
  (塊C 已驗證此行為)。這兩支的呼叫端是 Stripe webhook(`stripeWebhook.ts`,系統事件驅動),
  **沒有人類 actor / 沒有 ctx.user**,呼叫 audit() 等於「看起來有留痕、實際沒寫入」。
- 這兩處的寫入本身已可追溯:`deferStripeBookingIncome` 寫 `trustDeferredIncome` 列
  (帶 matchMethod / notes / sentinel bankTransactionId),`reverseDeferral` 設
  `reversedAt` + reason 欄。加上 Stripe webhook 自身的 idempotency 表(stripeWebhookIdempotency),
  資金動作有完整 event → row 的軌跡,不是黑箱。
- 建議:若要「系統 actor 審計」,該做一個 `systemAudit()`(不需 ctx.user、寫一個
  system-actor 列)當作 F2 統一項,套用到所有 webhook-driven 錢寫入,而不是只在這兩支
  硬塞一個會 no-op 的 audit()。本批不做,交指揮裁。
