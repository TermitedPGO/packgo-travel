### #3 真修 + prod 探真(指揮打回釘現狀後補做)

**flyctl ssh 唯讀探 prod(packgo-travel)結論**:`bankTransactions` 共 1612 筆
(408 入帳 / 1204 出帳),「stripe」字樣在**任何欄位**(merchantName / description /
originalDescription / paymentMeta)出現次數 = **0**。PACK&GO 真實金流處理商入帳是
**Square**(descriptor 形如 `ACH CREDIT Square Inc SQ ON 06/01`),不是 Stripe;
Stripe 目前只做線上 Checkout,其撥款尚未落進 Plaid 同步的銀行資料。

→ 後果:舊版「裸 stripe 單字即命中」在 prod 抓到 **0 筆真撥款**,卻會把任何姓名/
memo 含獨立單字 "stripe" 的真客人入帳誤判成撥款(轉撥不計收入)→ 該筆真收入靜默
從損益表消失、且永不進待認領。這是指揮說的認領漏斗漏洞(Ann 同類病),不是 F2。

→ 修法(不對稱風險取捨,按真資料定):`isStripePayoutInflow` 改為要求
`hasWord("stripe")` **且** 至少一個撥款語境 token(`payout`/`transfer`,
Stripe 預設 ACH statement descriptor 為 "STRIPE TRANSFER"/"STRIPE PAYOUT")。
裸 stripe 不再命中 → 落 pending_claim 交 Jeff 覆核(安全:漏抓變人工看,不是靜默
消失)。日後真 Stripe 撥款開始落地,Jeff 看到真實 descriptor 再校準
`STRIPE_PAYOUT_CONTEXT_TOKENS`(按真資料定,不憑空猜)。

證據:
- `accountingKnowledge.test.ts`:兩條原「釘現狀 true」翻成 `toBe(false)`
  (`zelle payment from stripe wong` / `ach from ann for stripe trip deposit`),
  補真撥款形狀 `toBe(true)`(STRIPE TRANSFER / STRIPE PAYOUT / payout: stripe.);
  補 `STRIPE_PAYOUT_CONTEXT_TOKENS` 清單斷言。
- `bankTransactionLinkEngine.process.test.ts`:新增整合例「客人姓 Stripe 的入帳 →
  processInboundTransaction 分派 pending_claim」(不再被誤 link stripe_payout)。
- `stripePayoutDeclassifyBackfill.test.ts`:受影響 fixture(裸 STRIPE / Stripe Inc)
  改成帶語境的真撥款形狀(STRIPE TRANSFER / STRIPE PAYOUT),12 tests 續綠。

⚠ **順帶暴露的 F2 議題(不在本批修,交指揮)**:真實入帳有 Square 撥款
(`ACH CREDIT Square Inc SQ`,INCOME_CONTRACTOR)。若 Square 是 Checkout 處理商,
Square 撥款落地同樣有 stripe_payout 式的雙計/漏斗風險,但塊B/C 只處理 Stripe。
Square 的遞延/雙計防護建議進 F2。

### webhook auditLog 豁免申報(回爐 #4)

`deferStripeBookingIncome` 與 `reverseDeferral`(trustDeferralService.ts)兩處錢寫入
**確實沒有呼叫 `_core/auditLog.ts` 的 `audit()`**。監工給的兩個選項擇一,選「T6 申報豁免」,理由:

- `audit()` 硬性要求 `ctx.user`(admin 已登入 context),沒有就 `log.warn` 靜默 return
  (塊C 已驗證此行為)。這兩支的呼叫端是 Stripe webhook(`stripeWebhook.ts`,系統事件驅動),
  **沒有人類 actor / 沒有 ctx.user**,呼叫 audit() 等於「看起來有留痕、實際沒寫入」。
- 這兩處的寫入本身已可追溯:`deferStripeBookingIncome` 寫 `trustDeferredIncome` 列
  (帶 matchMethod / notes / sentinel bankTransactionId),`reverseDeferral` 設
  `reversedAt` + reason 欄。加上 Stripe webhook 自身的 idempotency 表(stripeWebhookIdempotency),
  資金動作有完整 event → row 的軌跡,不是黑箱。
- 建議:若要「系統 actor 審計」,該做一個 `systemAudit()`(不需 ctx.user、寫一個
  system-actor 列)當作 F2 統一項,套用到所有 webhook-driven 錢寫入,而不是只在這兩支
  硬塞一個會 no-op 的 audit()。本批不做,交指揮裁。

**指揮批准豁免的條件(2026-07-09 驗收裁示,記進本補遺)**:
1. **F2 派工單含強制項 `systemAudit()`**:系統行為者(LOCAL_SCRIPT_TOKEN 端點 +
   webhook)的留痕機制,屆時 `deferStripeBookingIncome` / `reverseDeferral` /
   sandbox cleanup `confirm` **三處全接**,不再是豁免。
2. **sandbox confirm 日後執行時,輸出全文貼 `progress.md` 當追溯**:在 systemAudit()
   建好前,sandbox confirm(刪 First Platypus 帳戶 + 交易)這個一次性動作的完整
   端點回應(deletedAccounts / deletedTransactions / accountIds)要原樣貼進
   `docs/features/finance-dept/progress.md`,補足這次沒有 DB 審計列的追溯缺口。
