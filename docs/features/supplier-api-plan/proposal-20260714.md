## 五、統一 SupplierAdapter 介面(設計,待 Codex 複核)

```ts
interface SupplierAdapter {
  readonly code: "lion" | "uv" | ...;
  capabilities(): CapabilityMap; // 每能力: "live"|"cached"|"manual"|"unsupported"
  browseTours(q): Promise<NormalizedProduct[]>;        // 兩家可 live/cached
  getDepartures(productCode): Promise<NormalizedDeparture[]>;
  revalidateQuote(ref): Promise<QuoteResult>;          // UV live 已親證;Lion 需補接
  holdInventory(ref): Promise<HoldResult | Unsupported>;      // 兩家現況 unsupported
  createBooking(packet): Promise<BookingResult | Unsupported>; // 兩家現況 unsupported
  getBookingStatus(ref): Promise<StatusResult | Unsupported>;
  cancelBooking(ref): Promise<CancelResult | Unsupported>;
}
```
- unsupported 是明確回傳值非例外假裝;前台/訂單流程按 capabilities() 分流,絕不知道供應商欄位差異。
- 落地策略:先把既有 lion/uv client 收斂進 adapter 殼(browse/departures/revalidate 三支真實作,其餘 unsupported),NormalizedProduct 從「定義未用」變「唯一過境格式」;供應商官方文件到手後才逐支開洞。
- 與既有資產關係:supplierSync shared.ts 執行殼保留;supplierProducts/Departures 鏡射層不動;缺的 FK(鏡射↔零售)列入 adapter 批前置。
