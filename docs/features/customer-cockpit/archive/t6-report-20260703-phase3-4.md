## 3. 偏離申報

- 3a:sourceInteractionId 原本設計要「recordOutboundEmailInteraction 回傳新建 id」,實作階段發現這個函式只回 {recorded: boolean},選擇先用「查最新一筆」的次選方案——這個選擇後來被審查抓出是真的 race condition,已在修復階段從根解決(讓函式改回傳 insertId),等於偏離最終被自我修正回原始設計意圖,過程留作記錄。customOrderId 固定傳 null(escalationBox 這條回信路徑本來就沒有現成的 customOrderId 可用,設計文件本身也講「承諾是客人層級」,不算偏離)。
- 3b:runInquiryAgent 的輸入組裝方式,設計文件只講「用既有資料組出輸入」沒有指定細節,實作追蹤了 gmailPipeline.ts 的真實呼叫點照抄同一套組法;policyRules 刻意不傳(讓函式退回內建預設政策),因為 eval 只關心草稿品質不是即時政策,是刻意簡化不是疏漏。
- 4:design 文件原本只講「每項都要帶 customerProfileId」,實作發現前端既有選客機制對註冊客人用的是 userId、訪客才用 profileId(兩個不同 id 空間),所以在 TodayListItem 多加了一個 userId 欄位——這是為了讓功能真的能正確跳轉,必要的最小延伸,已在報告與程式碼註解說明。

## 4. 已知限制

- 3a:未處理「同一 profileId 短時間內連續寄兩封信」的並發窗口本身(修復拿掉了「查最新一筆」的猜測,但如果 Jeff 真的短時間連續寄兩封信,兩次呼叫仍是各自獨立的請求,理論上第二次的 insertId 一定對應第二封信,這條路徑審查後其實已經解決,不是殘留限制,列在這裡是為了誠實交代審查過程中出現過的疑慮)。
- 3b:eval-history.md 用固定格式的一行文字(**綜合分:X.X/10**)當機器可解析的來源,不是結構化資料庫,格式如果之後被人手動改壞,parseLastMonthScore 會靜默回 null(不做劣化比較)而不是報錯——這是刻意的簡化(寧可少一次比較,也不要把文件寫入邏輯複雜化成資料庫等級)。
- 4:承諾未兌現查詢目前無分頁上限,一人公司資料量下沒有效能疑慮,若客人/訂單量大幅成長需要重新評估;查詢失敗與「今天真的沒待辦」在 UI 上呈現相同(靜默降級),跟 2a/2c 既有看門狗面板同一慣例,不是這次新增的設計缺口。

## 5. 給指揮的審查建議(自曝弱點,優先看這幾點)

1. **3a 的 mark_promise 工具怎麼跟既有工具鏈搭配這件事,我在派工單裡自己也承認不確定**——Jeff 要在聊天裡說「這個承諾已經兌現了」,LLM 要先知道 promiseId 才能呼叫 mark_promise,但目前沒有一個「列出這位客人所有未兌現承諾」的既有查詢工具給 LLM 先呼叫拿到 promiseId。這條路徑在真實對話裡走不走得通,沒有做過端對端驗證(本機無 DB/LLM),建議這是驗收時第一個要實測的地方。
2. **3b 的「3 個獨立評審」獨立性,只做到了 code 層面的獨立(各自呼叫、不共用對話歷史)**,但沒有驗證這 3 次呼叫在同一個模型上是否真的會給出有意義的差異化判斷(如果 Opus 對同一個 prompt 高度收斂,3 次呼叫可能高度雷同,削弱了「多評審獨立性」原本想達到的效果)。這個問題本質上是模型行為層面,審查 workflow 沒辦法用讀程式碼的方式驗證,只能等第一次真實跑月度評分後看實際輸出的分數分佈判斷。
3. **4 的日期換算修復是這批最關鍵的一條**——修好之前,todayList 的三條規則(報價過期/出發倒數/尾款到期)每天都會因為 UTC vs LA 時區差而有系統性偏差(不是隨機 bug,是每天在特定時間窗口都會錯)。這類錯誤特別陰險,因為在多數測試時段(白天)看起來完全正常,只有在太平洋時間傍晚後才會出現。這次審查抓到了,但這提醒了一件事:凡是跟「今天」有關的邊界計算,以後每次新增規則都要把這個檢查當成標準流程的一部分,而不是等審查抓。
