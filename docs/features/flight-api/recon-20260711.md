# 機票 API 供應商偵察報告

> 任務:把 PACK&GO 的機票流程從「人工 Trip.com 查訂 + 業主親自刷卡」升級成「系統內搜尋、鎖位、業主按付款、自動出票」。
> 性質:純研究,不寫程式、不開帳號。
> 偵察日期:2026-07-11。所有資訊皆附來源連結與查證日期;查不實者標「未確認」,沒有編造。
> 適用主體:PACK&GO, LLC(加州一人旅行社,**無 IATA / ARC 牌照**;客群主力為華語美中、美台、美日國際線)。

---

## 0. 一句話結論

**主線推薦 = Duffel(API 自動化骨幹)+ 一家美亞線淨價 consolidator(GTT Global / Sky Bird / Centrav 擇一,做深折扣覆蓋與人工後盾)。**
Duffel 一家就把「無牌照、美國出發、搜尋→鎖位→業主確認→扣款→自動出票、退改走 API、自己加價」全部打通,而且是 pay-as-you-go 無月費無保證金;它唯一的弱點是美亞線傳統航司的「鎖位延遲付款」是逐航司開放、且深折扣淨價未必最漂亮,所以搭一家專做美中/美台/美日的美國 consolidator 補價格與覆蓋。Trip.com 公開只能「搜 + 導流」不能訂,Kiwi 技術上最貼合但目前邀請制進不去,Amadeus Self-Service 便宜但不能自己出票、仍要綁 consolidator,Travelport/Sabre 直連對一人公司門檻過高。

## 1. 背景與約束

| 項目 | 現況 / 約束 |
|------|------------|
| 牌照 | 無 IATA、無 ARC。任何方案第一關都是「沒牌照能不能出票」。|
| 現行流程 | 人工上 Trip.com 查價與下單,業主 Jeff 親自刷卡付款。|
| 目標流程 | 系統內搜尋比價 → 鎖位(hold)→ 業主按下付款 → 自動出票。|
| 硬紅線 | AI 絕不動錢、不碰卡號;付款一律業主親自觸發(對齊現有 checkout fail-closed 停止線)。|
| 客群 | 華語客群為主,美國出發往中國、台灣、日本(United、ANA、EVA、China Airlines、Air China 等)。|
| 加州法規 | 加州 Seller of Travel(CST)已註冊;賣機票仍受 CST 規範。信託帳(Trust #5442)/ 營運帳(#2174)的機票入帳方式需業主與會計確認(見 §7 註記),本報告不代決。|

底線:公開產品**只能搜尋+導流**,不符合「自家系統內鎖位→業主確認→自動出票」。可留作**比價參考 + 聯盟第二收入**(呼應 memory 的 affiliate 養第二收入),不能當骨幹。唯一能訂的路是簽鎖住的分銷 API,而且 Trip.com 仍是出票 consolidator、非你。

## 4. 推薦路線與理由

**主線:Duffel 為自動化骨幹 + 一家美亞線 consolidator 為價格/覆蓋後盾(GTT Global 優先,Centrav 為政策透明備選)。**

理由:

| 需求 | Duffel 怎麼滿足 | consolidator 補什麼 |
|------|----------------|--------------------|
| 無牌照出票 | Managed Content 靠 Duffel 美國牌照,直接可出 | consolidator 自持牌,plating 深折扣淨價 |
| 系統內搜尋 | Offer Requests API,一次整合 NDC+GDS+LCC | 補 Duffel 未必最漂亮的美亞線淨價 |
| 鎖位→業主確認→扣款 | 原生 Hold + Pay later,三支獨立 API | GTT 可建 PNR 交出票,等同人工 hold |
| 退改 | 全 API | 人工 ops(含中文),複雜 case 後盾 |
| 自己加價 | API 內 markup 全拿 | 淨價你加 markup 或賺公價佣金 |
| 成本可控 | pay-as-you-go 無月費無保證金 | 押金錢包制(簽約前確認額度) |

為何不選其他為主線:Amadeus 不能自出票(Duffel 已涵蓋搜尋)、Travelport/Sabre 直連門檻過高、Kiwi 邀請制進不去且虛擬聯程風險、Trip.com 公開不能訂。

Trip.com **保留**做兩件小事:比價交叉參考、聯盟連結賺第二收入(不進出票骨幹)。

風險備援:Duffel 的 ADM 分攤條款與美亞線 hold 開放度、consolidator 的押金隔離與非 IATA 受理,都是**簽約前書面確認項**(見 §8)。

## 5. 分階段接入計畫草案

先搜尋比價 → 再鎖位 → 最後出票,每階段先 Duffel sandbox 驗證,不先寫正式碼。

| 階段 | 目標 | 動作(系統) | AI 停在哪等業主 |
|------|------|------------|----------------|
| **P0 評估(現在)** | 選定供應商、驗覆蓋與價格 | 開 Duffel sandbox;同步開 GTT / Centrav 帳號洽談;拿核心航線(SFO/LAX ↔ TPE/PVG/NRT)跑覆蓋與比價,對照現行 Trip.com 價 | 全程只讀,不涉錢。業主看比對表決定是否進 P1 |
| **P1 搜尋比價** | 後台/客戶頁能搜、能出比較 | 接 Duffel Offer Requests,只讀顯示;可人工並列 consolidator 淨價報價 | AI 只搬運航班/票價資訊(呼應「搬運不生成」),不自動出價、不建單 |
| **P2 鎖位 hold** | 對合格 offer 建 hold、顯示到期倒數 | 對 `requires_instant_payment:false` 的 offer 建 `type:"hold"`(不帶 payments);系統把 `price_guarantee_expires_at`、`payment_required_by` 呈現給業主;**hold 不扣款** | 系統只鎖位不扣款;不合格(需即時付款)一律 **fail-closed 擋下轉人工**(對齊現有 checkout 停止線) |
| **P3 出票 capture** | 業主按鍵才扣款出票 | 業主點「確認付款」→ 系統呼叫 Duffel payment 端點;付款走 Duffel Payments(代收旅客卡,PCI 元件,AI/業主都不手打卡號)或業主先儲值的 Duffel 餘額 | **只有業主按鍵才呼叫 payment;AI 永不觸發扣款**。付款前系統重取單驗價,遇 `schedule_changed` 停手回報 |

漸進原則:P1 上線穩了再開 P2;P2 的 hold 行為驗證足夠(尤其美亞線哪些航司給 hold)再開 P3。任一階段 tsc 綠 + 測試綠才 commit。

## 6. 「AI 絕不動錢」流程設計

對齊硬紅線與現有 fail-closed 停止線(commit d212294 的即時結帳 fail-closed 模式)。

| 環節 | AI/系統做到哪 | 誰動手 | fail-closed 規則 |
|------|--------------|--------|-----------------|
| 搜尋 | AI 搜、彙整、出比較表 | 系統自動 | 資訊 100% 搬運不生成;查不到就標未確認 |
| 報價草稿 | AI 產客人報價(套 packgo-quote/flight-ticket skill) | 系統自動 | 單價一律以供應商後台/API 即時價為準,不採 flyer 文字(呼應報價紅線) |
| 鎖位 hold | 系統建 hold(**不扣款**)、顯示到期倒數 | 系統自動 | 只對 `requires_instant_payment:false`;需即時付款者擋下轉人工 |
| **扣款/出票** | 系統把「待業主確認付款」排到業主面前 | **業主 Jeff 親自按鍵** | **AI 永不呼叫 payment 端點**;無業主點擊=不扣款 |
| 卡號 | 一律 Duffel Payments PCI 元件收旅客卡 | 旅客自輸 / 系統元件 | AI 與業主都不手打卡號(對齊「Claude 不碰卡號」硬線) |
| 退款/改期 | AI 備妥取消/改期報價與 `refund_amount` | 業主確認才執行 | 退款是動錢,業主點頭才送出 |
| 稽核 | 每次扣款/出票留 log + 對 Trust/Operating 入帳提示 | 系統記錄,業主/會計覆核 | 見 §7 入帳註記 |

一句話:**AI 可以把單子準備到「差業主按一下付款」為止,錢的那一下永遠是業主的手。** 鎖位不扣款,所以 hold 由系統做不違紅線;真正扣款只走業主親自觸發的按鈕。

## 7. 入帳(Trust)註記,需業主與會計確認

機票款與旅遊團訂金的信託會計不同。旅遊團訂金明確走 Trust #5442、出發後才 recognize(CST §17550)。機票款多為「代航司收取後轉付」的 pass-through,未必比照團訂金入 Trust。**本報告不代決**,列為業主 + 會計確認項:機票收款進哪個帳、何時 recognize、Duffel Payments 代收模式下金流落點。附帶:加州 CST 對「賣航空運輸」有專門規範,PACK&GO 已註冊 CST,新增機票線前確認涵蓋範圍即可。
