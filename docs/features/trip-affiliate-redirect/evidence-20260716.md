# Trip.com affiliate link verification — raw evidence
Date: 2026-07-16. Method: read-only HTTP GET (curl) + in-app browser landing check.
No login, no cookies of Jeff's, no order submission, no customer data, zero payment.

Test data only: SFO→NRT 2026-09-15/22 (2 adults); Tokyo 2026-09-15/18 (1 room, 2 adults).

## A. The `/t/{materialId}` path format used by ALL current production code is DEAD

| URL (as generated by server/services/affiliateLinkService.ts) | Result |
|---|---|
| `https://www.trip.com/t/D13390057?allianceId=..&sid=..&dcity=SFO&acity=NRT&ddate=..&rdate=..&adult=2` | `302 → location: /` (entire query string dropped) → `302 → https://us.trip.com/?locale=en-us` → 200. **No Union cookie.** |
| `https://www.trip.com/t/D15196722?allianceId=..&sid=..&city=Tokyo&checkin=..&checkout=..` | same: 302 → `/`, params dropped, **no Union cookie** |
| `https://www.trip.com/t/D13390050?allianceId=..&sid=..` | same: 302 → `/`, params dropped, **no Union cookie** |

Set-Cookie on first hop of `/t/D13390057` = only `ibulanguage`, `ibulocale`,
`cookiePricesDisplayed`, `UBT_VID` (generic visitor id). No affiliate attribution.

| `https://www.trip.com/?Allianceid=..&SID=..&trip_sub3=D13390050` | 200 → us.trip.com, params preserved, **Union SET** |

## C. CONTROLLED EXPERIMENT — the shape is the bug, not the material IDs

`D13390050` is the material that appears in Jeff's approved entry (that is the full claim —
not "valid" and not "commission-effective": the table below shows the cookie cannot
prove material validity).

| Form | Union cookie |
|---|---|
| `?trip_sub3=D13390050` (query fmt, approved-entry material) | **SET ✓** |
| `?trip_sub3=D13390057` (query fmt, flights ID) | **SET ✓** |
| `?trip_sub3=D99999999` (query fmt, **BOGUS** ID) | **SET ✓** ← trip_sub3 is an unvalidated passthrough label |
| `/t/D13390050` (path fmt, approved-entry material) | **NOT SET ✗** |

Conclusion: the same approved-entry material shows attribution-cookie behaviour in query form and none in
`/t/` path form. Therefore `/t/{materialId}` is not a valid Trip.com affiliate link
shape. Also: a Union cookie does NOT validate a material ID (bogus ID still sets it),
so cookie-set ≠ material valid ≠ commission payable.

## D. Deep-link search params — do they land?

FLIGHT `https://www.trip.com/flights/showfarefirst?dcity=sfo&acity=nrt&ddate=2026-09-15&rdate=2026-09-22&triptype=rt&class=y&quantity=2&Allianceid=..&SID=..&trip_sub3=S14595667`
- Browser: **Union cookie SET** (set by JS — curl on first response showed none, so
  the curl-only check was a FALSE NEGATIVE; browser check was required).
- Search executed: "37 flights found", real SFO→NRT results (ZIPAIR SFO 17:15 → NRT
  20:00, US$642), dates "Tue, Sep 15 — Tue, Sep 22", "2 adults · Economy". PASS.
- CAVEAT: the "Going to" destination input renders **empty** (`value: ""`) even though
  results are correctly SFO→NRT. Cosmetic/UX defect on Trip.com's side.

HOTEL `https://www.trip.com/hotels/list?city=228&checkin=2026-09-15&checkout=2026-09-18&Allianceid=..&SID=..&trip_sub3=S18716875`
- **Union SET ✓**, city=228 resolves to Tokyo, "6,291 properties found",
  "Tue, Sep 15 - Fri, Sep 18, 3 nights", "1 room, 2 adults, 0 children". Full PASS.
- Requires a **numeric** Trip.com city ID.

## E. HOTEL city-name param is IGNORED — wrong-city risk (customer-facing)

Current code sends `city=Tokyo` (a NAME). Discriminating test:
requested `city=Osaka` → page rendered **Tokyo** (search box "Tokyo", 6,291 properties,
Ikebukuro/Asakusa hotels). The city name is ignored and Trip.com falls back to the
visitor's remembered last search.

Implication: a customer clicking "Osaka" could land on Tokyo results. An earlier
`city=Tokyo` test appeared to pass — that was **contamination** from a prior `city=228`
visit, caught only by testing an unvisited city.
