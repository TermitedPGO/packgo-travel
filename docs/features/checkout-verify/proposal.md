# checkout-verify — 結帳驗位與揭露存證

> Stage 1 需求。2026-07-11 立(外部顧問第二輪審計 §二/§三;Jeff 裁決:貨架放量前下訂模式定型,本批為線三硬前置)。

## 要解什麼

現況結帳即請款:付款前不驗即時位價、供應商確認在收款後人工進行、沒有保存客戶付款前同意的文件版本。四種曝險(審計 §二):客戶已付款但供應商可能無位/價變;Jeff 收款後救火;Stripe 費用與 chargeback 成本;無揭露版本難證明客戶同意內容。

## 為誰

- 客人:不會再付了錢才發現沒位/價變。
- Jeff:退款救火時間歸零方向;爭議時有「客戶同意版本」存證(配 CA B&P §17550)。

## 範圍(本批 = 三模式框架的模式一)

1. 結帳前即時驗證(UV):建 Stripe Session 前 live 驗 (a) 在售 (b) 有位 (c) 價格容差 $0 + 必付清單一致 (d) 記錄資料新鮮度。任一失敗不建 Session,回結構化錯誤轉詢位。
2. 付款前揭露存證:Session 建立前持久化「客戶即將同意的版本」快照(團名/班期/單價人數/必付明細/條款文字/幣別/驗證結果與時間);webhook 完成時釘死關聯。
3. 詢位 fallback:驗證失敗或非 UV 商品 → 前端轉「提交訂位需求」進既有 inquiry 流。
4. 非 UV 商品一律擋即時請款(fail-closed 硬邊界)。

不在本批:模式二(授權後確認 capture_method=manual)、模式三(詢位後付款連結)、可展示/可索引/可收款三態商品狀態。

## 驗收長怎樣

- 驗證四項各自失敗 → Session 不建、零 Stripe 呼叫(紅綠測試釘死)。
- 價格不一致(canonical 整數 USD 基準上任何差異)也擋。
- UV API 逾時/炸 → fail-closed 擋轉詢位,絕不 fail-open 收錢。
- 驗證通過 → Session 建立且快照落庫欄位齊;webhook 蓋章關聯。
- 既有 bookings/stripeWebhook 測試不弱化。
- 觀測:驗證失敗率、擋單原因分佈進結構化 log。
