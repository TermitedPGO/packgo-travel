## 三件 Jeff 沒問、但我認為最重要的事

### 1. 在這個環境裡，內容比 code 危險

Code 有 guard 鏈：tsc、vitest、deploy guard、code review。內容沒有 guard：你寫給客人的航班時間、飯店設施、價格，聽起來永遠是對的，錯了也不會有紅字。PACK&GO 的邊界原則「AI 出手處資訊 100% 正確：搬運不生成」是唯一防線。執行判準：客人會看到的每個事實，你要能指出搬運來源（供應商後台、API 回應、Jeff 的原話、官網查證）。指不出來源的句子，要嘛去查，要嘛刪掉。歷史教訓：David 案漏比對 invoice 差點把成本價出給客人；Excel 公式壞掉產出 NaN 報價。工具會壞、文件會騙，來源鏈不會。

### 2. 工具堆積是無聲的稅，也是誘惑

這個環境掛著大量 MCP（Adobe 80+ 工具、Canva、Drive、Chrome、computer-use...）和 100+ skills。兩個後果：system prompt 固定變重；更危險的是「工具在手就想用」。你的判準：任務沒點名要用的花俏工具，不用。出 PDF 走 packgo-* skill 的既定流程，不因為看到 Adobe MCP 就改道。給 Jeff 的建議（每季提醒他一次）：把非常用 MCP 斷開，要用再連，他自己的 memory 也記過這件事。

### 3. docs/features/ 有 30 個資料夾，大多是歷史不是現況

判斷一份 feature 文件還算不算數：看 progress.md 的最後日期 + `git log -3 -- docs/features/<name>/`。超過一個月沒動的，當歷史讀，不當指令執行。接手進行中的工作，以最近的交接段（本檔附錄）和 git log 為準，不要考古舊 proposal 然後把已放棄的方案做出來。

## 這套制度最可能的退化方式（與預防）

1. 規則通脹：每次踩坑加一條規則，半年後 CLAUDE.md 又回到 400 行，弱模型重新抓不到重點。預防：加規則前先問「這能不能寫成既有判準下的一個新例子？」加例子永遠優於加規則；40-maintenance.md §4 的行數閾值是硬的。
2. 驗證形式化：fresh 驗收退化成「派了就算」，subagent 回 PASS 沒人看證據。預防：驗收回報必須逐條附證據（20-judgment.md J2），主對話抽查其中一條的真偽；抽查抓到假 PASS 就是 P0 事故，寫回教訓。
3. 制度與 code 脫節：路徑改了、指令換了，standards 沒跟上，之後的模型按過時文件做錯。預防：發現文件錯，修文件就是當前任務的一部分（綠區直接改），不是「以後再說」；40-maintenance.md §7 健康檢查。
4. 例外累積成先例：「這次趕時間就不寫測試 / 直接 deploy」。一次例外 = 永久先例，下個弱模型會引用它。預防：例外只有 Jeff 本人能開，開了要在對應檔案記一行（誰、何時、為什麼）；你自己永遠沒有開例外的權限。
5. 交接斷裂：session 突然中斷，下一個 session 花兩小時考古。預防：長任務每完成一個批次就把狀態寫進 feature 的 progress.md（隨做隨寫）；被迫中斷時用下方附錄格式留交接。

## 誠實條款（Fable 的極限聲明，也是你的）

這套制度能補執行品質：拆解、驗證、多樣本評審，Sonnet 等級跑得動。它補不了兩樣東西：
- 品味與模糊題：設計方向、語氣、取捨。按 20-judgment.md J6 做（多候選給 Jeff 挑 / 對照基準 / 直說做不到）。不要假裝。
- 未知的未知：制度只覆蓋已踩過的坑。遇到制度沒寫的情況，回到第一性：這個動作可逆嗎？出公司門嗎？碰錢嗎？有來源嗎？四問裡有任何一個答案不確定，就停下來問。

制度是地板不是天花板。它保證你不會犯我們已經犯過的錯；新的錯，靠你把它寫回來（40-maintenance.md §3），讓下一個你少踩一個坑。
