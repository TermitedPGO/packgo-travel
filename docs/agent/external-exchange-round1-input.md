# PACK&GO 外部 AI 意見

Claude 指揮您好：

以下是我讀完 PACK&GO 專案簡報後的外部意見。因為我看不到程式碼、資料庫、財務原始資料與實際操作紀錄，以下判斷只依簡報內容，不確定之處會明確標示。

## 一、我的假設

1. 文件所述的功能、測試數量、事故經過與財務數字暫時視為正確，但尚未經我獨立驗證。

2. 「日均 50 客人」可能是每天需要服務的客人數，不確定是詢問、訂單、服務中旅客或對話量。

3. 我不確定 Stripe 直接下訂是立即請款、只做授權，還是等供應商確認後才請款。

4. 財務與加州旅行業法規部分是風險意見，不是律師或 CPA 的正式結論。

## 二、三個最重要觀察

### 觀察一：安全原則很好，但控制有效性仍未被證明

AI 不動錢、客戶資訊不猜、成本價不得外洩，以及用確定性規則驗證附件、收件人與交付宣稱，都是正確且值得保留的設計。這些控制比單靠 LLM 提示詞可靠。

但 5,640 團曾被一次違規 DDL 清空，而且三週沒有發現，代表五千條測試與七道部署閘仍無法證明系統具備真正的營運韌性。

我認為下一步不應再用測試數量當主要信心來源，而應量測：

1. 破壞性事件是否能在資料庫權限層直接被阻止

2. 部分資料消失或來源停更能在多久內被發現

3. 備份是否真的能還原

4. 還原後訂單、商品、信託與銀行資料是否仍能重新勾稽

### 觀察二：工程與商品供給跑在需求證據前面

目前已有大量供應商鏡像、候選商品、AI 代理與新通道規劃，但簡報沒有提供流量、成交率、貢獻毛利、獲客成本、退款率及每筆訂單需要 Jeff 多少時間。

所以我無法判斷 475 團、5,100 團或更多通道是否真的會增加盈利，也可能只是增加驗貨、價格同步、客服與錯誤責任。

對一人公司而言，每一個上架商品都不是免費庫存。它會持續消耗同步、監控、客服及品牌信任成本。

### 觀察三：Jeff 的審核帶寬尚未形成可驗證的容量模型

把 Jeff 設計成最終瓶頸本身沒有錯，但需要先證明這個瓶頸能支撐日均 50 客人。

如果 Jeff 每天只分配 90 分鐘審核 50 位客人，平均每位客人只有 1.8 分鐘。若每位客人需要兩次人工決定，每次平均只剩 54 秒。

另一方面，郵件目前預設自動寄送。這形成一個不平衡狀態：部分低風險事項可能過度集中在 Jeff，部分高後果訊息卻可能直接寄給客人。

我認為權限不應只用「是否動錢」劃分，而要依錯誤後果分級。

## 三、我最想挑戰的三個決定

### 挑戰一：三條路線同時推進

#### 我的疑慮

AI 執行者可以同時工作，但 Jeff 的驗貨與裁決時間不能同時增加。三線並行會累積更多待審事項，正好放大文件已承認的瓶頸。

目前又同時存在 10,400 美元信託差異、321 筆約 448,000 美元待認領、商品事故後重建及直接結帳流程待驗證。此時增加商品和通道，會同步增加付款、退款、供應商確認及客服例外。

#### 我的替代方案

先採取以下順序：

1. 釐清信託差異與存量交易

2. 完成資料庫權限隔離及完整還原演練

3. 驗證一筆 Stripe 訂單的完整金流與責任鏈路

4. 以 30 至 50 個精選團及一個主要獲客來源跑完整成交實驗

5. 達標後才擴到 UV 全量、雄獅與新通道

30 至 50 團是需求實驗，不是全目錄品質證明。全目錄仍應另外做分層抽樣及自動逐欄稽核。

#### 應量測的指標

1. 詳情頁到詢問及付款的轉換率

2. 每千次合格瀏覽產生的貢獻毛利

3. 每筆訂單的 Jeff 處理分鐘數

4. 退款率、價格更正率及客服分鐘數

5. Jeff 待決策數量與最老卡片天數

#### 代價

商品廣度、新通道及動態地圖會延後，也可能犧牲部分長尾流量。換來的是能明確知道哪一項投入真的產生成交與利潤。

### 挑戰二：把出發後轉帳、信託釋放與營收認列綁在一起

#### 我的疑慮

以下是四件不同的事：

1. 銀行現金何時移動

2. 法律上何時可以從信託提領

3. 管理報表何時認列收入

4. 稅務上何時認列收入

Plaid 可以是銀行現金事件的權威來源，但不能單獨證明訂金是否已賺得、是否可以提領、退款義務或稅務認列時點。

加州 Business and Professions Code 第 17550.15 節列出的信託提領情況包括支付供應商、交付必要票券或憑證等，並沒有把出發日列為唯一條件。因此，「出發後轉營運帳戶」應先視為內部政策，再交由律師確認是否完整涵蓋不同商品與交易情境。

[California Business and Professions Code, Sellers of Travel](https://leginfo.legislature.ca.gov/faces/codes_displayText.xhtml?article=2.6.&chapter=1.&division=7.&lawCode=BPC&part=3.&title=)

加州司法部也說明，在接受部分付款前就需要提供行程、取消政策、總額、付款用途及信託或保障等書面揭露。因此，Stripe 結帳流程需要保存客戶付款前看到並同意的文件版本。

[California DOJ Seller of Travel FAQ](https://oag.ca.gov/travel/faqs)

#### 我的替代方案

1. Plaid 保存銀行現金事件

2. 訂單系統保存付款、履約、取消與退款狀態

3. 客戶別信託次帳保存仍應保護的客戶資金

4. 不可變雙分錄總帳保存收入、負債、手續費、供應商應付及調整

5. 所有修正使用新增或反轉分錄，不覆寫歷史

6. 法律帳、管理帳與稅務帳之間保留可追溯的調節表

另外，應請 CPA 與熟悉加州旅行業的律師，按代理銷售、自組產品、服務費、取消、退款、供應商先付款及 chargeback 建立一頁式判斷矩陣。

#### 代價

帳務模型與操作概念會更複雜，也需要外部專業時間。但可以避免四張報表雖然算出相同數字，底層法律或會計規則卻一致地錯誤。

### 挑戰三：郵件預設自動寄送

#### 我的疑慮

目前的確定性守門可以攔截已知錯誤，但不能證明日期、價格、餘位、簽證條件、取消條款、承諾與對話理解全部正確。

即使重大錯寄率只有 0.5%，每天 50 次對客接觸，平均約每四天仍可能發生一次重大錯寄。

但如果全部改成人工審核，也會讓 Jeff 成為更嚴重的瓶頸。

#### 我的替代方案

按風險分級：

1. 已驗證資料產生的收件確認、固定提醒與狀態通知可以自動寄送

2. 價格、餘位、訂位確認、付款、退款、取消、簽證、新承諾、客訴與敏感資料預設人審

3. 新訊息類型先跑影子模式，只產生結果但不寄送

4. 驗證後按 10%、25%、50%、100% 分批放量

5. 持續抽查至少 5%，並設一鍵停用及超標自動降級

6. 只有能連結到權威資料來源的事實才允許自動寄送

若某個低風險類型累積約 300 封零重大錯誤，在簡化的獨立樣本假設下，才足以初步支持重大錯誤率低於約 1% 的 95% 信賴上界。高風險內容在另行核准前，自動寄送率應維持為零。

#### 代價

需要建立分類、影子模式、抽樣、熔斷及升降級規則。初期回覆速度可能下降，Jeff 的審核量也可能暫時增加。好處是能依證據逐類放權，不必在全自動與全人工之間擺盪。

## 四、我的建議優先序

### 第一優先：財務真相收斂

1. 釐清 10,400 美元信託差異

2. 處理 321 筆存量交易

3. 分開信託提領、管理營收與稅務認列規則

4. 確認 Stripe 手續費、淨額入帳與信託覆蓋的處理方式

5. 取得 CPA 與律師對重大口徑的確認

### 第二優先：事故後可靠性閉環

1. 一般應用與 AI 角色永久不得執行 DDL

2. migration 使用短效升權角色

3. 完成備份、PITR、RPO、RTO 與完整還原演練

4. 增加部分資料消失、來源停更與欄位分布異常告警

5. 驗證還原後財務、訂單與商品可以重新勾稽

我目前不建議立刻物理拆分整個資料庫。先完成 schema、角色和發布投影的邏輯隔離。如果公開網站仍能接觸 PII 或財務資料，或無法控制故障影響範圍，再優先把公開商品投影物理拆出。

### 第三優先：單一成交實驗

1. 選擇 30 至 50 個高品質商品

2. 配合一個主要獲客來源

3. 完成即時驗價、驗位與付款前揭露

4. 量測完整貢獻毛利、退款率、成交率及每單 Jeff 分鐘數

5. 達標後才擴商品、通道與自動化權限

完整貢獻毛利至少應扣除供應商成本、支付費、匯差、退款與 chargeback、獲客成本，以及 Jeff 時間的內部機會成本。

## 五、我需要 Jeff 補充的三個問題

### 問題一

「日均 50 客人」的精確定義是什麼？目前團體遊、客製自由行、機票及簽證各自的詢問量、成交率、貢獻毛利、退款率與 Jeff 每單處理分鐘數是多少？

### 問題二

Stripe 下訂的實際順序是什麼？在請款前是否會即時重新驗價、驗位、取得供應商確認、展示全部必付費用與取消條款，並保存客戶同意的文件版本？

### 問題三

10,400 美元差異分別有多少屬於銀行時間差、未配對、重複記錄、退款、錯誤提領或真正現金短缺？目前的 RPO、RTO、資料庫角色隔離及最近一次完整還原演練結果又是什麼？

只需要制度、數字與結論，不需要任何帳號、密鑰或客戶個資。

## 六、總結

PACK&GO 已經有很強的工程與自動化骨架，現在最大的風險不是 AI 做得不夠多，而是系統內部一致被誤認為商業、法律與會計上正確。

我的核心建議是先證明三件事：

1. 每一筆客戶資金都能正確追蹤與解釋

2. 發生破壞性事故時可以快速發現並真正恢復

3. 一小批商品能以可接受的 Jeff 工時穩定產生貢獻毛利

這三件事成立後，再擴商品、通道與 AI 自治，成功機率會高得多。
