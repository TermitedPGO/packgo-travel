# 證據與營運規範（治理單一事實源，2026-07-12 立）

> CLAUDE.md 只放 2-3 行摘要與路由,細節在此。本檔由 Codex 第 5.5 輪審查採納的治理控制彙整,指揮維護。改本檔前讀 40-maintenance。

## 一、狀態語言階梯（完成宣稱只准走到證據支持的那一階）

規劃中 → 已施工 → 已合併 main → 已部署 → 已啟用（旗標 ON）→ 正式驗證（部署後 smoke/postcondition 過）→ 穩定運行。

- commit 只代表「已提交」,不代表已合併/已部署/已啟用/正式驗證/穩定。
- 旗標未開的功能只能標「已部署」,不能標「已啟用」。
- 中斷的批次視為未發生,不得引用其預期產出。

## 二、evidence_reference 鐵則

任何完成宣稱必附非空 evidence_reference,指向實際產物：檔案路徑+行號、測試輸出、prod 唯讀查詢輸出、部署後 smoke、供應商/銀行/處理商導出。派工單、計畫文件、agent 自述、「測試全綠」「另一 agent 同意」都不算證據。

## 三、三層驗證（高風險交付）

1. fresh review：查範圍、推理、漏項。
2. 確定性不變量：系統必須拒絕或必須成立的條件（測試釘死）。
3. 外部真相/正式 postcondition：供應商回應、銀行/處理商導出、真實 DB 錯誤碼、部署後 smoke。

高風險完成至少到第 2 層;涉正式外部事實需到第 3 層。fresh agent 是第二雙眼睛,不是獨立真相源（可能與施工者共享同一錯誤前提）。

## 四、高風險類別（命中任一即走完整工作流,不看行數）

錢、帳、客戶承諾、PII、schema、權限、部署、對外訊息、批次資料操作。五行 migration/三行權限變更/一行付款映射比 300 行純 UI 更危險。

## 五、WIP 上限（指揮佇列限流）

正式施工主題 ≤2、高風險批 ≤1、證據就緒等驗收 ≤2、等待 Jeff ≤3、每批 Jeff 決策 ≤3、單批預估審查 ≤30 分鐘。唯讀掃描/研究/文件不占施工名額,但待審研究產物 ≤3。指令、STATE、測試、正式事實衝突時停手回報,不挑最方便的一份。

## 六、正式環境資料庫紀律（P0,6-17 事故預防層）

- 已核實（2026-07-12 SHOW GRANTS）：prod runtime 身分現為等同 root（含 DROP/CREATE/ALTER/CREATE USER/SUPER）。硬化前這是活風險。
- 一般 session 不得直接修改正式資料;runtime 身分禁 DDL。schema 只走 tracked migration + 短效 migration 身分。
- 正式診斷預設唯讀、最小權限、核准腳本、留 audit reference。
- 重抓/回填/破壞性驗證/資料修復先在隔離 clone;DDL 實測只打 canary schema（無客戶資料）套同 grants,意外成功即停列 P0。
- 五種權限分開對待:唯讀查核 / 受控寫入 / migration / 回填 / 還原演練。不再用「DB 操作」一詞包成一類。

## 七、信託與金流（法律定性歸律師/CPA,工程只存可證事實）

- 客戶旅遊款項具信託責任,不得僅以出發日推導可提領或可認列。銀行現金移動、法律可提領、管理營收認列、稅務認列是四件不同的事。
- 每筆信託轉出須綁定旅客/訂單、法定提領類型、金額、原始證據、Jeff 核准。矩陣未定或證據不足即停手回報。
- 收款框架是「全部進帳」,非五通道（五通道僅佔真實收款約 29%,見 channel-aggregate）。

## 八、證據安全（銀行/處理商/客戶原始資料）

不進 Git、不進 iCloud/Dropbox/OneDrive 等同步路徑、不進任何 AI 對話。保全目錄用受控非同步位置（現為 ~/Documents/PACKGO_evidence_preservation_*）,原始檔唯讀,分析用工作副本。manifest 只記來源/期間/筆數/hash/缺口/安全位置,不抄帳號個資,副本與原始分開。

## 九、工位 preflight（iCloud 事故預防層）

repo 不得放 iCloud/Dropbox/OneDrive 同步路徑（現於 ~/dev/網站）。node_modules/build cache/worktree 不進雲端同步。新工位啟用先跑路徑檢查 + git fsck。GitHub 只保護已 push 內容,未 commit/未 push 的本機工作不受保護。

## 十、客戶身分合併（多渠道前置）

email/phone 先正規化,命中視為「候選」非自動同一人;一對多、欄位衝突、高風險操作轉人工。channel identity 用渠道原生 stable ID + 唯一約束;新增/連結用 transaction 或 DB constraint 防競態,留完整稽核。

## 十一、高風險事故第一次就建預防層

資料刪除、金錢、PII、公開錯價、錯誤客戶承諾:第一次發生即建機械控制（CI/lint/權限/守門）,不等錯第二次。教訓只有「守門」或「例子」兩種合法形態;T6 收尾必答「教訓轉成了哪個守門」。
