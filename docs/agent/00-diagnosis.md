# Harness 診斷：三大洩漏與修法

> 2026-07-02 由 Fable 5 立制 session 寫成。基於實測證據，非印象。
> 後續檔案引用編號：D1（長對話自己下場）、D2（開場固定稅）、D3（自寫自驗）。
> 讀者是未來的主模型（Sonnet / Opus / Haiku）。每條修法都設計成你做得到的動作。

---

## D1. 最大洩漏：長對話不切 + 指揮官自己下場讀

### 證據
- 本機 122 個 session transcript 共 2.0GB。最大單一 session 576MB，另有三個 100MB+。
- CLAUDE.md §9.6 早有「session > 80 turns 換對話」紅線，transcript 證明長期被違反。
- Jeff 的 memory 已記錄：token limit 燒快的主因是長對話的 cache reads。

### 機制（為什麼燒錢）
每讀進主對話的一個工具輸出（整個檔案、grep 大結果、網頁全文），之後**每一輪**都要再付一次它的 cache read 費；prompt cache TTL 只有 5 分鐘，一過就付全額。一個 200 輪的 session，第 10 輪讀的 2000 行檔案，等於被重複計費 190 次。

### 修法（可執行判準）
1. **讀取三行規則**：預期輸出超過 ~100 行的讀取（掃 repo、多檔搜索、網頁全文、log 全文），一律派 `Explore` subagent，只回結論與 `檔案:行號`。主對話只允許讀：即將要 Edit 的目標段落、subagent 回報、單一小檔（<150 行）。
2. **任務邊界 = session 邊界**：一個 feature 批次收完（commit + 回寫 progress.md）就結束 session。下一批用文件交接開新 session。判準：對話裡出現第二個不相關主題、或你發現自己在往回捲找三十輪前的內容，就該切了。
3. **長產物不進對話**：任何 >50 行的產出（報告、清單、草稿）直接 Write 到檔案，對話裡只留路徑 + 三行摘要。

## D2. 第二洩漏：開場固定稅過重、記憶索引誘導失焦

### 證據
- 舊 CLAUDE.md 21KB（439 行）+ MEMORY.md 索引 12KB（73 條記憶、60+ 索引行）= 每個 session 開場固定載入 33KB，約 12k~15k tokens。
- 索引每行都有 hook 句，弱模型容易被不相關條目誘導去 recall 全文（例如做前端任務卻讀了機票報價 memory）。
- 舊 CLAUDE.md 混雜歷史敘事（「2026-05-24 加，三次踩坑教訓」段落），對執行沒有指令價值。

### 機制
開場稅是乘法：33KB × 每一輪 cache read × 每個 session。且內容越長，弱模型抓錯重點的機率越高（規範埋在故事裡，紅線和軼事同樣字重）。

### 修法
1. **CLAUDE.md 已於 2026-07-02 重寫為路由檔**（本次立制）：硬紅線全文在場，細節抽到 `docs/standards/`，按任務類型索引載入。原則：CLAUDE.md 只放「每個 session 都必須知道」的內容；「某類任務才需要」的內容進 standards 檔並被路由表指到。
2. **記憶索引只在命中時 recall**：看 MEMORY.md 索引時，先問「這條和當前任務的名詞有交集嗎」，沒有就不讀 body。禁止「先全部讀一遍以防萬一」。
3. **記憶膨脹閾值**：MEMORY.md 索引超過 80 行時，跑 `anthropic-skills:consolidate-memory` skill 合併同主題條目（規則見 40-maintenance.md）。

## D3. 第三洩漏（出錯主因）：自己寫自己驗 + 錯了原地重試

### 證據
- git 歷史顯示所有 code 由主模型自寫自驗（一人公司無 reviewer）。多個 commit 是在修「上一個 session 自稱完成但其實壞的」東西（例：草稿誠實度三 gate、scorecard 對帳抓出的六修）。
- CLAUDE.md §9.4 實測教訓已記錄：progress.md 自稱「全部完成可上線」，並行驗證仍抓到 1 個 P0 + 3 個漂移測試。**文件說 ready ≠ 真的 ready，自己說 ready 更 ≠ ready。**
- 弱模型的失敗模式：同一個錯誤原地重試三、四次，每次只改表面（改個參數再跑），token 燒掉、方向沒變。

### 機制
寫作者驗自己的產出，會用寫作時的同一套假設去驗，盲點原樣保留。模型越弱，這個效應越強。原地重試則是把「執行失敗」誤讀成「執行不夠用力」，但多數失敗是方向錯。

### 修法
1. **驗收一律 fresh-context**：完成宣告前，派一個乾淨 subagent 按驗收條件驗（檔案用 read-back、程式用測試或實跑）。詳見 `10-dispatch.md` 的驗證不自驗節。
2. **兩次規則**：同一子任務同一方法失敗 2 次，禁止第 3 次原樣重試。必選一條：換方法、升級模型（帶完整失敗軌跡）、或停下報告。判準與範例見 `20-judgment.md`。
3. **高風險判斷加第二意見**：出給客人的內容、財務數字、部署決策，用多答案評審（生成 2~3 個候選 + 一個 fresh judge 挑選）而不是單發。

## 誠實條款：這套修法補不了的

- **模糊題與品味判斷**：拆解與驗證能保住執行品質，但「這個設計方向對不對」「這封信的語氣夠不夠暖」這類判斷，弱模型的天花板就是天花板。遇到時：(a) 產 2~3 個差異明顯的候選讓 Jeff 挑，不要自己選一個交卷；(b) 或明說「這題超出我的判斷力，建議你自己看」。假裝有把握是最差選項。
- **長鏈推理的隱性錯誤**：多步推導（報價計算、日期換算鏈、法規適用）弱模型會在中間步驟無聲出錯。修法是把推導外化成可驗的中間產物（每步寫出來、用工具算、逐日加總驗算），不是「更小心」。
- **本診斷的時效**:基於 2026-07 的 harness。cache 機制、工具、模型陣容變了就要重驗，見 `40-maintenance.md`。
