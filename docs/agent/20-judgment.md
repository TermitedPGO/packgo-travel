# 判斷力外化：五個關鍵時刻的判準

> 讀者：未來的主模型。這些判準把「經驗直覺」翻成你可以機械執行的規則。
> 每條格式：判準 → 正例（該這樣）→ 反例（不該這樣）。例子全部來自 PACK&GO 真實場景。

## J1. 何時升級模型

判準（任一命中就升級，具體路徑見 10-dispatch.md §6）：
- 任務需要同時考慮 3 個以上互相牽制的約束（例：改 schema 同時要顧 migration、既有查詢、加密層、前端型別）
- 兩次嘗試後錯誤「變形」但沒消失（第一次 A 錯、修完變 B 錯）
- 任務涉及紅線（CLAUDE.md 二節的 8 條）且你對做法沒有把握
- 需要從矛盾的資料中裁決（兩份文件說法不同、測試和文檔打架）

正例：sonnet 修「回信路由到錯的 Gmail 帳號」，改了兩輪，第一輪修好單帳號但壞了轉寄場景。此時停手，整理兩輪軌跡升 opus。這正是多約束互牽的型態。

反例：haiku 批次改檔名改錯一個，於是把整批任務升 opus 重做。錯。改錯的那一個用 sonnet 重做即可，模式沒錯就不用升級整批。升級的單位是「卡住的子任務」，不是整個任務。

## J2. 何時算真的完成

判準（全部滿足才准說「完成」）：
1. 驗收條件逐條有證據：每條寫得出「怎麼驗的 + 結果」。寫不出來的那條 = 沒完成
2. 代碼類：`tsc --noEmit` 0 錯 + 相關測試綠 + 新邏輯有新測試
3. 重要交付（出給客人、碰錢、上 main 的 feature）：fresh subagent 驗收過（10-dispatch.md §7）
4. 文檔回寫：`docs/features/<name>/progress.md` 反映實際狀態
5. 交付物裡沒有未聲明的 TODO / 佔位 / 「暫時先這樣」。代碼註解裡的 TODO 若已列在回報的「未做」清單裡就算已聲明；沒列的 = 未聲明 = 沒完成
6. 給 Jeff 的回報包含：做了什麼、沒做什麼、怎麼驗的、已知風險

正例：cockpit 草稿功能修完，回報寫「三個 gate 都有測試（server/followupDrafts.test.ts 新增 4 case 全綠）、tsc 0 錯、英文客人場景用 0909 測試帳號實測過、沒改到的已知問題：抬頭偶爾重複，已記進 progress.md」。

反例：「已修復，應該可以了」。「應該」= 沒驗。2026-05-29 bookkeeping 教訓：progress.md 自稱全部完成，獨立驗證抓到 P0。自稱完成不是完成，驗證通過才是。

## J3. 何時停下來問 Jeff

判準：先分類再決定。
- 直接做（不問）：可逆、在原需求字面範圍內、不出公司門。改 code、跑測試、寫文檔、commit（green 即 commit 是 Jeff 授權的）
- 先問（用 AskUserQuestion，帶選項帶你的建議）：
  a. 需求有兩種以上合理讀法，選錯會浪費半天以上（例：「把報價整理一下」是重排版還是重算價？）
  b. 不可逆或對外：寄信給客人、刪資料、對外發布、花錢
  c. 發現需求本身可能是錯的（Jeff 說改 X，但 X 的現狀跟他描述的不一樣）：先報告發現，不要照做也不要自作主張改成 Y
  d. 品味裁決（J6）
- 問的紀律：一批問完（最多 4 題），每題附預設選項和你的推薦。禁止擠牙膏式一次問一題
- 無法問的環境（headless / 非互動 session，沒有 AskUserQuestion 工具）：停下，把問題、選項、你的推薦寫成最終回報後結束。不要帶著不確定繼續做

正例：Jeff 說「幫連先生把行程改到 7/16 出發」，但檔案顯示 7/16 那班機票價已變。停，報告差異，問要鎖價還是改期。這是 c 類：現實和需求的前提衝突。

反例：任務是改按鈕圓角，因為「不確定要 rounded-lg 還是 rounded-xl」跑去問。錯。這在 `docs/standards/design.md` 表格裡有答案（按鈕 = rounded-lg）。有規範可查的事不問。問之前先查：CLAUDE.md 路由表 → standards 檔 → memory 索引 → repo 內文檔。

## J4. 方向錯了的訊號（換路，不是再試一次）

任一訊號出現，停止當前方法。「更用力再試一次」不在選項裡：
1. 同一個錯誤訊息在換了修法之後仍原樣出現（你修的不是病因所在的層）
2. 打地鼠：修 A 壞 B，修 B 壞 C（抽象層選錯了，該往上一層看）
3. 每前進一步都要加一個新的特判 / hack（架構在抵抗你，方向錯）
4. 你開始想繞過測試、lint、guard，而不是滿足它們（guard 擋你通常是 guard 對）
5. 範圍膨脹：原估 3 個檔案，實際摸到第 8 個還沒完（理解錯了問題的形狀）

換路的動作（按序考慮）：回到上一層重新診斷（用 5 Whys 找真正的層）→ 換一條實作路線 → 升級模型帶完整軌跡 → 問 Jeff。同一件事最多 2 輪重試，第 3 輪必須是上述四者之一。

正例：修「摘要日期錯誤」，第一輪改 prompt 沒好，第二輪發現根因是沒把今天日期餵給 LLM（grounding 缺失），換路修資料流而不是繼續調 prompt 措辭。

反例：測試紅了，把測試的期望值改成實際輸出讓它變綠。這是訊號 4 的典型違規：測試在告訴你 code 錯了，你卻讓測試閉嘴。

## J5. 品質底線怎麼驗（按交付類型）

- 代碼：tsc + vitest + fresh review（高風險加 packgo-code-reviewer）。UI 改動用 preview 工具實看，不憑 class 名腦補效果
- 數字（進客人文件或帳務的）：每個數字要有來源鏈（供應商後台模擬訂單 / API 回應 / DB query 結果），並用第二條路徑驗算（Excel 公式壞過：芝加哥案 Day4 NaN，逐日加總驗算才抓到）。沒有來源的數字不出門
- 客人文件：用對應 packgo-* skill 產出，不手寫 HTML；出檔前逐項比對供應商 invoice 確認成本價沒上文件；細節（貴賓室、設施、招牌菜）上網查實，禁止通用佔位文字
- 報告 / 分析：每個結論要能指到證據（檔案:行號、URL、query 結果）。指不到的寫「推測」，不寫成事實
- 檔案操作：批次操作先在 1 個樣本上跑通再全量；產新版前刪舊版（桌面不堆版本）

## J6. 模糊題與品味題（誠實條款的執行）

弱模型（包括你）在這類題上天花板有限：設計方向、文案語氣、「這樣夠不夠好」。制度補不了品味，只能補流程：
1. 首選：產 2~3 個差異明顯的候選（不是同一方案的三個微調），真實渲染（HTML/截圖），讓 Jeff 挑。Jeff 明確偏好真實渲染而非線框
2. 有基準就對照基準：路線圖以 v357 捷克/奧地利極簡風為準；客人文件以黑白極簡為準；Jeff 說「弄清楚」是放大原圖不是重新設計
3. 都不適用時，直說：「這題需要品味裁決，我給出候選但建議你自己看。」假裝有把握然後交一個自選答案，是最差選項

正例：新頁面設計，產 3 個真實 HTML 版型（保守 / 大膽 / 折衷）附真照片給 Jeff 挑。

反例：自己選了「最現代的」設計直接實作完交付。Jeff 的極簡黑白是刻意 art direction，「現代化美化」大概率被整組打回，浪費整輪。

## J7. 風險分級速查（決定驗證強度與是否先問）

- 高風險（fresh 驗收 + 需要時第二意見；對外動作先問）：出給客人的任何東西、碰錢碰帳、刪資料、schema 變更、部署相關、寄信發布
- 中風險（fresh 驗收）：上 main 的 feature code、批次改檔、自動化規則變更
- 低風險（自驗即可）：文檔、scratchpad 實驗、read-only 查詢、單檔小修

- 2026-07-14 反例:執行者依派工單在共用工作樹切 feature 分支,指揮同時段 commit docs 落錯分支(4e4cfeb6),事後 reset 復原。規則化:執行者佔用工作樹施工(尤其涉分支切換)期間,指揮禁 commit/禁切分支;需並行時派工加 isolation: "worktree"。
