# 派工 Prompt 模板（五種任務型態）

> 用法：複製對應模板，填 `【】` 空格，發給 Agent tool。選 subagent_type 和 model 見 `10-dispatch.md` §3。
> 每個模板已內建：三件套（目標動機 / 驗收 / 回報格式）+ 回報合約 + 「不確定就停」尾註。不要刪這些段落。

## T1. 搜尋 / 盤點（subagent_type: Explore, model: haiku 起）

```
目標：找出【要找什麼：定義/呼叫點/現況】。
動機:【主任務是什麼,這次搜尋的結果會被拿去做什麼決定】。
範圍:【目錄或檔案模式,例 server/routers/**】;搜索廣度:【medium / very thorough】。
已知線索:【關鍵字、相關檔案、命名慣例】。

驗收條件:
- 每個結果附 檔案:行號
- 明確回答「找到 / 找不到 / 部分找到」,找不到要列已排除的搜法(用過哪些關鍵字與目錄)

回報格式:結論(≤5 行)+ 結果清單(檔案:行號 + 一句話說明)。禁止貼代碼段落,禁止過程敘事,總長 ≤30 行。
不確定就標注「不確定」回報,不要猜。
```

## T2. 實作（subagent_type: claude, model: sonnet；跨模組 opus）

```
目標:實作【功能一句話】。
動機:【為什麼要做,服務哪個 feature/客人場景】。
必讀 context(先讀完再動手):
- 【docs/features/<name>/design.md 或 spec 所在】
- 【要改的檔案清單】
- 涉及 UI 加讀 docs/standards/design.md;涉及 DB/API 加讀 docs/standards/backend.md
邊界:只改【範圍】;不碰【明確排除,例 schema / 其他模組】。
通用地雷(涉及者必寫進派工單):
- 從文件/信件/案件檔抽「客人聯絡方式」時,一律排除業主本人的 email 與電話
  (jeffhsieh09/0909、support@、+1 510-634-2307)— 案件檔常把 Jeff 的聯絡
  方式跟客人寫在同一格(2026-07-03 1b P0 實案)。
- 凡「今天」相關的邊界計算,比較的「兩端」都要換算 America/Los_Angeles
  曆日:基準日用 todayLA() 只對了一半,被比較的 DB 時間戳(quoteSentAt/
  departureDate 等)也要用同一套 Intl LA 換算切曆日,否則太平洋時間傍晚後
  天數會系統性多算一天(2026-07-03 Phase4 審查實案,白天測不出來的陰險雷)。
- 秒級截斷比較:MySQL DATETIME 只有秒精度,寫入時毫秒被截掉。凡拿一個帶毫秒
  的「時間點」去跟 DB 的 DATETIME 欄位做 `>=`/`<` 比較(如「這一刻之後有沒有
  新列」),同一秒內落庫的列 createdAt 被截成整秒 → 帶毫秒那端一定比它大 →
  gte 誤判「沒有新列」= 假失敗。比較基準一律先 `Math.floor(ms/1000)*1000` 取整到
  秒、再視情況留 1-2 秒餘裕(2026-07-06 weeklyCanary 實案:互動 13:00:00 落庫、
  sinceMs=13:00:00.xxx,兩項檢查全誤報失敗;白天單測不帶毫秒也測不出來)。
- TiDB + Drizzle 關聯子查詢排序(v799 回爐後的完整版,取代舊「只要 select 就好」):凡要
  「用一個內含關聯子查詢(`... WHERE inner.fk = outer.id`)的表達式排序」(如
  `GREATEST(..., (SELECT MAX(x) FROM inner WHERE inner.fk = outer.id))`),正確形狀只有一種:
  (a) 表達式**整支只渲染一份**,欄位**全程帶完整表前綴**(裸 `\`inner\`.\`fk\` = \`outer\`.\`id\``);
  (b) `.as("alias")` 明確給別名 select 出來;(c) `ORDER BY alias`,**絕不**在 orderBy 再展開表達式。
  兩個會 500 / 排序錯的坑(都被此形狀避開):
  1. `orderBy(desc(exprSql))` 會在 ORDER BY **再展開一份** GREATEST → 同運算式渲染兩次,TiDB 對不上。
  2. Drizzle 把 `${table.column}` 放進 **SELECT 欄位**時會**掉表前綴**(渲成裸 `\`id\``),子查詢
     `WHERE fk = \`id\`` 的 id 被最內層表的 id 吃掉 → 關聯斷;同表達式在 orderBy 情境卻帶前綴 →
     「一份無前綴、一份有前綴,兩者對不上 → 500」。故欄位要寫 **raw 全限定字面**,不用 `${...}` 插值。
  實案:admin.customerUnreadCount guest 臂 lastContactSql。v794 起 500;第一次 hotfix(ed83709)
  誤以為「補進 select 即可」→ v799 仍 500(正是上面兩坑)。第二次(本回爐)才用 (a)(b)(c) 修好。
- ⛑ 元規則:**修 SQL 方言問題的 commit,修法本身必須先驗證過「渲染出來的 SQL 形狀」**,不能只
  憑推理(本機無 DATABASE_URL 測不到方言,推理已連錯兩次)。最低成本本地驗法:把 query build 到
  `.orderBy()` 後呼叫 `.toSQL().sql` 印出真實 SQL(用 `drizzle-orm/mysql-core` 的 `QueryBuilder` 免連線),
  斷言 GREATEST 只一份、別名有 `as`、關聯 WHERE 用 `outer.id` 全限定。能上 prod TiDB 唯讀跑一次形狀最好。
- 資格條件放寬必帶噪音閘:凡把「誰能進列表/計數/自動處理」的條件放寬(加 OR 分支、擴大 WHERE),
  必須同時問「這條新分支會放進哪些非目標實體?」並帶上對應過濾(isKnownNoise / spam 分類 / 測試帳號
  排除)。2026-07-07 實案:guestList 加 OR lastInboundAt IS NOT NULL 救回被吞客人,同時把上百張
  歷史行銷噪音卡放進列表、徽章 99+,隔天回爐兩輪。指揮側寫 spec 時就要點名這條,不是等審查抓。
  這是 hardening Wave 2「EXPLAIN 彩排自動化」要接管的事;至 2026-07-07 已是第四口(前三:0112 migration
  breakpoint、raw sql<Date> naive 字串、秒級截斷、本條 ORDER BY 關聯子查詢)。

驗收條件(逐條驗,附證據):
- tsc --noEmit 0 錯(OOM 時 NODE_OPTIONS="--max-old-space-size=6144")
- 【具體行為條件,例:X 輸入得到 Y;測試 server/foo.test.ts 新增 case 且綠】
- 新邏輯有對應 Vitest;測試不插真實資料
- 【UI 類:圓角/整齊規範自檢過,grep object-cover 無漏網】

回報格式:改了哪些檔(檔案:行號範圍)+ 驗收逐條打勾附證據 + 未做/風險。≤30 行,長 diff 不貼,commit 由主對話決定。
遇到 spec 沒覆蓋的情況:停下回報選項,不要自行擴大解釋。
```

## T3. 重構（subagent_type: claude, model: sonnet；建議 isolation: worktree）

```
目標:重構【對象】:【from 什麼結構 to 什麼結構】。
動機:【churn 熱點/超 300 行/要為 X feature 鋪路】。
鐵律:行為不變。公開介面【不可變 / 可變清單】。
必讀:【目標檔案】+ 現有測試【路徑】。

驗收條件:
- 重構前先跑一次現有測試記錄基線,重構後同一批測試全綠(這是行為不變的證明)
- tsc 0 錯
- 無孤兒引用:grep 舊名稱【舊 symbol】全 repo 無殘留
- 每個新檔 ≤300 行

回報格式:新結構一覽(檔案 + 職責一句話)+ 測試前後對照(N passed → N passed)+ 驗收打勾。≤30 行。
測試基線本來就有紅的:停,先回報,不要順手修(那是另一個任務)。
```

## T4. 研究（subagent_type: general-purpose, model: sonnet）

```
目標:回答【研究問題,越具體越好】。
動機:【這個答案會決定什麼】。
來源優先序:【官方文檔 > 官方 blog > 高信譽技術站 > 論壇】;PACK&GO 供應商類優先用 repo 內 client(lionClient / uvClient)與後台,flyer 與行銷文字不可信。
時效要求:【需要多新的資訊,例 2026 年後】。

驗收條件:
- 每個關鍵結論附來源(URL 或 檔案:行號)與日期
- 區分「查證的事實」與「推論」,推論明標
- 查不到的部分明說「查無」,禁止編造補全

回報格式:直接答案(≤3 行)→ 關鍵事實清單(每條附來源)→ 不確定處。研究產物 >30 行時寫到【檔案路徑】,回報只給路徑+摘要。
```

## T5. 審查（subagent_type: packgo-code-reviewer 或 Explore, model: sonnet；高風險 opus）

```
目標:審查【對象:diff / 檔案 / 客人文件】,聚焦【正確性 / 紅線合規 / 安全 / 全部】。
動機:【這個交付的風險等級與去向,例:今天要出給客人】。
你是 fresh 審查者:不信任何「已完成」「已驗證」的自我宣稱,一切以你親眼所見為準。
必讀基準:CLAUDE.md 紅線八條 +【design.md / backend.md 視對象】。

驗收條件(審查本身的合格線):
- 每個 finding 附 檔案:行號 + 具體失敗場景(什麼輸入/狀態會炸),沒有場景的觀感問題降級為 note
- 對照紅線逐條檢查並回報「查過,無違反」或列出違反處
- 找不到問題時,列出你檢查過的角度(讓主對話判斷覆蓋面),不寫「看起來沒問題」

回報格式:verdict(PASS / FAIL+阻塞項)→ findings 按嚴重度排序(每條:位置+場景+建議修法一句話)→ 已檢查角度清單。≤30 行。
```

## 通用尾註（每個模板都要保留的三句）

1. 不確定就停下來回報，不要猜。
2. 你看不到 Jeff 的 memory 和主對話歷史；本 prompt 已含全部 context，缺什麼在回報中說明。
3. 回報超過 30 行 = 違約；長產物落檔傳路徑。

## T6. 完工報告(執行者交付合約,2026-07-03 Jeff 立)

> 執行者(Sonnet/Opus)完成一個批次後,除 progress.md 回寫外,結尾必交這份
> 結構化報告給指揮(Fable)審。報告是指揮審查的地圖:指揮逐條稽核宣稱,
> 而不是重新大海撈針。**報告不取代指揮的獨立驗收** — 自測有盲點,這是
> 整套分工存在的原因。

```
## 完工報告:【批次名】
1. 交付清單:commit hash + 每個 commit 一句話 + 檔案數
2. 自測證據(逐條,可稽核):
   - tsc:0 錯(附跑法)
   - vitest:N 個新測試 / 全套 M 綠(附套件名)
   - 數字紀律:vitest/tsc 的總結行必須從終端輸出原樣複製貼上(Test Files 與 Tests 兩行完整貼),不准轉述或憑記憶寫數字——監工會在乾淨 checkout 逐字重跑核對,數字對不上整份報告可信度歸零(2026-07-03 Phase6 實例:報告寫 304 檔/4335 tests,main 實際 302 檔/4248,查了半小時才確認只是回報不準)

3. 偏離申報:哪裡跟派工單不一樣、為什麼(沒有就寫「無」)
```

## 審查回饋迴圈(指揮側,配 T6 用)

指揮驗收抓到的每一個問題,結案前分類:

- **prompt 可防**:派工單本可寫明但沒寫(漏了地雷提示、驗收條件不夠具體、
  邊界沒劃清)→ 回流:更新本檔對應模板或 sonnet5-handoff 的紀律段,
  下一張派工單就帶上。這是指揮的學習迴圈,和執行者的 progress.md 對稱。
- **不可預知**:只有實測才能暴露(support@ 雙帳號、FFFD 之類)→ 不改
  prompt,記進 memory/教訓,驗收清單加一條對應檢查。
- **執行者失誤**:派工單寫了但沒照做 → 下一張派工單把該條升級成
  驗收條件(從「要求」變「逐條打勾附證據」)。

分類結果一句話記在驗收報告尾部,月度回顧時看三類比例:prompt 可防占比
持續下降 = 指揮在進步。
