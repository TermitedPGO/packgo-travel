> 安全邊界:本文件不含任何秘密。token、API key、銀行帳號、客人個資一律不進本檔,交流中也不提供。
> 維護:指揮(Claude)每次重大批次收尾更新;版本日期見文末。

## 一、公司與目標

PACK&GO 是一人旅行社(業主 Jeff,加州 Newark),主營華語客群的美洲團體遊、客製自由行、代訂機票簽證。目標:用 AI 把一人公司撐到日均 50 客人的服務量,瓶頸設計在「Jeff 的審核帶寬」而不是人力執行。

經營原則(等同憲法):
1. 自動化優先,萬不得已才人力,品質與公平不可犧牲。
2. AI 絕不動錢:認領、認列、退款、轉帳、付款全部只有 Jeff 能按。
3. 供應商成本價絕不出現在客人文件;報價先在供應商後台核對真價。
4. 客人看到的資訊要嘛 100% 正確,要嘛留白,禁止猜測生成。

## 二、產品全景

整個專案是一個網站兩張臉:對客的公開旅遊網站,和 Jeff 一個人用的後台作業系統,共用同一個 codebase 與資料庫。

### A. 客人面:packgoplay.com 公開網站

- 頁面:首頁、/tours 貨架(篩選目的地/天數/價格)、/tours/:id 行程詳情(逐日行程、景點、餐宿、費用三分類「團費含/必付/自費」、未來班期與三級餘位、Stripe 直接下訂)、/custom-tours 客製需求表單。
- 語言:繁中預設 + 英文,i18n 約 7,900 key 雙語齊(有 parity 測試守門,JSX 禁止硬編碼中文)。
- 設計語言:黑白極簡骨架 + 彩色滿版 hero 賣景;所有可見元素圓角;像素對齊與密度節奏是紅線(業主的刻意 art direction,不是預設美學)。
- SEO:SPA + bot-prerender(搜尋引擎拿到完整快照,已實測 Googlebot 生效);路由白名單防直開 404。
- 商品資料流:供應商鏡像 → 零 LLM 重建管線(品質門檻:無價/無目的地/無行程/無班期一律不上架)→ 批次上架帶快照可回滾。圖片不用供應商圖(版權紅線),用 Unsplash 授權真照按目的地自動配 + 攝影師署名合規,冷門景點之後 AI 生圖補位。
- 現況:6/17 事故(一次違規 DDL 清空商品七表,5,640 團消失三週無告警)重建中;試批 16 團已 live,UV 全量約 475 團待業主驗貨放行;雄獅(台灣線約 5,100 團)接口已修待驗;行程頁翻修與每團動態路線圖(SVG 極簡風)排下一波。

### B. 業主面:/ops 後台(一人公司作業系統)

五域 + 首頁,高密度極簡設計(28 個舊分頁收斂成 5 域):
- /ops(指揮中心):審批卡片流(AI 產出的動作等 Jeff 一鍵批准/駁回)、客訴升級箱、收件匣。設計哲學:AI 停在哪等 Jeff,每頁先定義清楚。
- /ops/customers(客戶駕駛艙):每客一頁 —— 真實歸檔對話(Gmail 全套自動收發歸檔)、文件(PDF 解析)、承諾追蹤(從對話自動抽取「答應客人的事」)、漏價看門狗(成本/售價/確認書逾期自動叫)、AI chat(Jeff 問狀態、出草稿,讀真實對話不准猜)。
- /ops/tours(商品目錄):批次上架、校準、供應商同步。
- /ops/finance(財務駕駛艙):真相列四格(現金、待認領、信託在途、本月損益)+ 工作流(認領、認列、轉帳提醒)+ 月度損益/年終稅表(Schedule C 口徑)。
- /ops/marketing、/ops/settings。
- 另有 workspace 月報視圖與行動端。

### C. AI 代理群(server 端,LLM 只在 server 統一入口經快取)

- 收件流水線:Gmail 輪詢/推播 → 噪音過濾 → 客人配對 → 自動歸檔進客戶頁;詢價信自動分類 → 草稿產生 → 誠實守門(禁未經驗證的交付宣稱、禁稱呼不認識的收件人、宣稱附件必須真有附件,全部確定性規則零 LLM)→ 合規守門 → auto-send 閘(預設自動發,可疑進人工)。
- 跟進系統:安靜客人自動浮出、跟進草稿(暖語氣規範)、A/B 報告。
- 記帳代理:銀行交易 LLM 分類 + 確定性後衛(拿不準的類別強制進人工複核池,收入紀錄絕不自動中性化)。
- 財務警報:分類器 + 產生器 + 顧問(省稅建議方向,規劃中)。
- 退款代理:退款事件自動分診出摘要卡給 Jeff。
- 目錄品質:校準代理(內容忠實度/翻譯/圖/完整度打分)。
- 每週自檢:正確性稽核、金絲雀測試、信託勾稽看門狗(排程自動跑,異常出卡)。

### D. 財務系統(加州旅遊法合規)

- 權威帳:Plaid 銀行流水(信託帳戶與營運帳戶);次帳:手工 accountingEntries;五收款通道 Stripe/Square/Zelle/Venmo/PayPal。
- 信託合規(CST §17550):客人訂金進信託不算營收,出發後轉營運帳戶才認列。全生命週期:遞延 → 認列 → 轉帳偵測配對(單筆全等自動、群組只出建議)→ 每週勾稽。損益頁、月度趨勢、年終稅 CSV、財報四個口徑對同一筆訂金保證同數(有跨月紅綠測試)。
- 系統行為留痕:AI 或排程做的每筆帳務動作與人工操作同級進稽核日誌。

### E. 供應商整合

- UV(美洲線,公開門市 API 免登入,原生 USD):鏡像 7,600+ 產品每日同步,54,000+ 班期,成本價物理不存在於鏡像(零外洩面)。
- 雄獅 Lion(台灣/亞洲線,TWD 需換匯,同業價=成本絕不上客人頁,出口有禁字守門直接 throw)。
- 上架前做過 API 正確性審計:抽樣逐欄比對供應商即時 API 真值,售價 100% 一致才放行。

## 三、工程防線

- 測試約 5,000 條全綠才能部署;部署七道閘(branch/clean/sync/migration 清點/tsc/全套測試/一次性人身 token),只有 Jeff 能按;schema 變更只准 tracked migration 經 release 流程。
- SQL 彩排閘:部署前 238 條登記 SQL 在 prod 資料庫逐條 EXPLAIN,壞 SQL 上不去。
- 部署後煙霧八臂(含「對客團數歸零告警」,事故補課)、錯誤漏斗去重出卡、每週稽核。
- 資安:護照號加密存取(明文讀寫=事故)、PII 遮罩、客戶去重紅線(insert 前必查重)。

## 四、AI 組織運作(閉環)

- 指揮(Claude 高階模型):規劃、派工單、多路獨立驗收、裁決,不寫應用碼。
- 執行者(Claude 中階模型,背景 agent):按派工單施工,分支作業,交付自測證據。
- 每批交付由 fresh agent 多路審(重跑/正確性/範圍),P1/P2 回爐,過了才合主幹。實績:P1 級問題(如 LLM 分類器繞過人工複核的洞、必付費用被混進自費)都是驗收路抓出來的。
- Jeff 只保留三件事:部署按鈕、動錢點頭、品味裁決。決策晨晚兩班批量處理。

## 五、路線圖(三線並行)

## 七、與外部 AI 的交流協議

歡迎的輸入:策略第二意見(三線優先序、定價、獲客、自媒體)、架構批判(過度工程/欠防護/盲點)、合規風險(加州旅遊法、稅務、平台條款、版權)、產品視角(後台 UX、客人端轉化率、貨架呈現)。

規則:
1. 你(外部 AI)看不到程式碼與資料,只有本簡報。回覆先列假設再給意見;不確定就標明,不要編造本專案細節。
2. 建議帶取捨與代價,能量化就量化,不要只給結論。
3. 交流產物是「建議」不是指令:執行端(Claude 指揮)把它當輸入之一,逐點回應採納/反駁,分歧交 Jeff 裁決。
4. 不要向 Jeff 索取任何密鑰、帳號、客人個資。

交流方式:Jeff 人肉擺渡,把本簡報貼給外部 AI 開場,之後兩邊往返貼。
