# PACK&GO 專案簡報(外部 AI 交流版)

> 用途:交給外部 AI(ChatGPT 等)閱讀,取得第二意見。它沒有 repo 權限,本文件須自足。
> 安全邊界:本文件不含任何秘密。token、API key、銀行帳號、客人個資一律不進本檔,也不准在交流中提供。
> 維護:指揮(Claude)每次重大批次收尾更新;版本日期見文末。

## 一、公司與目標

PACK&GO 是一人旅行社(業主 Jeff,加州 Newark),主營華語客群的美洲團體遊、客製自由行、代訂機票簽證。目標:用 AI 把一人公司撐到日均 50 客人的服務量,瓶頸設計在「Jeff 的審核帶寬」而不是人力執行。

經營原則(等同憲法):
1. 自動化優先,萬不得已才人力,品質與公平不可犧牲。
2. AI 絕不動錢:認領、認列、退款、轉帳、付款全部只有 Jeff 能按。
3. 供應商成本價絕不出現在客人文件;報價先在供應商後台核對真價。
4. 客人看到的資訊要嘛 100% 正確,要嘛留白,禁止猜測生成。

## 二、系統概況

- 技術棧:React 19 + Tailwind 4 + tRPC 11 + Drizzle ORM + MySQL(TiDB)+ Redis/BullMQ,部署 Fly.io,網站 packgoplay.com。
- 部署紀律:只有 Jeff 能部署(pnpm ship,七道閘:branch/clean/sync/migration 清點/tsc/全套測試/一次性人身 token);schema 變更只准 tracked migration 經 release 流程,任何人對 prod 跑手動 DDL 屬違規(6/17 曾因此丟過整個商品目錄,已建告警防重演)。
- 測試文化:新功能必有紅綠測試,全套約 5,000 測;另有 SQL 彩排閘(部署前 238 條登記 SQL 在 prod 上逐條 EXPLAIN)。
- 資料分層:供應商鏡像層(supplierProducts/Details/Departures,7,600+ 筆)→ 策展層(tours,對客)→ 交易層(customOrders/payments)→ 財務層(bankTransactions 為權威損益,accountingEntries 為次帳)。

## 三、AI 組織運作模式(閉環)

- 指揮(Claude,高階模型):規劃、派工單、多路獨立驗收、裁決,不寫應用碼。
- 執行者(Claude 中階模型,背景 agent):按派工單施工,worktree 分支作業,pathspec commit,交付自測證據。
- 驗收:每批交付由獨立 fresh agent 多路審(重跑驗證、正確性、範圍與文件),抓到 P1/P2 回爐重修,過了才合 main。
- Jeff 只保留三件事:部署按鈕、動錢點頭、品味裁決(設計與文案的最終審美)。
- 決策節奏:晨晚兩班批量裁決,避免 Jeff 變成瓶頸。

## 四、三條業務線現況(2026-07-11)

線一 財務(基本收官,prod v808/v809):
- 信託合規:加州 CST §17550,客人訂金進信託帳戶不算營收,出發後轉營運帳戶才認列。系統全鏈:訂金遞延、出發認列、轉帳偵測配對、每週勾稽看門狗、損益/月度趨勢/年終稅表/財報四口徑對同一筆訂金同數。
- 財務駕駛艙(/ops/finance):真相列四格(現金、待認領、信託在途、本月損益)+ 待辦工作流;321 筆約 $448k 存量待 Jeff 分批認領。
- AI 記帳邊界:銀行交易自動分類,但 Square 撥款「故意不自動歸類」(查證發現撥款入帳是目前唯一收入紀錄,自動中性化=收入靜默消失),連 LLM 分類器都加了確定性後衛強制進人工複核。
- 未決:CPA 對認列口徑的答覆、信託帳實差約 $10.4k 的查核。

線二 客戶通道(已測繪,待動工):
- 現況:Gmail 全套自動收發歸檔已上線;客戶頁是駕駛艙(對話、文件、承諾、看門狗)。
- 波次計畫:0 內部訊息表合併 → 1 LINE(官方 API)→ 2 Meta 系 → 3 微信公眾號 → 4 iMessage 只收不發(Mac 讀本機資料庫,自動發送會被蘋果封號,不做)。

線三 公開網站(進行中,本週主線):
- 事故背景:6/17 一次違規 DDL 清空 tours 七表,5,640 團消失,三週無人察覺(已補歸零告警)。

基建:Windows 機當 24/7 常駐工位(WSL + Claude Code),MacBook 晚間回家補 iMessage 增量;Mac mini 暫緩。

## 五、與外部 AI 的交流協議

歡迎的輸入:
1. 策略第二意見:三線優先序、定價、獲客、自媒體漏斗、供應商談判。
2. 架構批判:上述設計哪裡過度工程、哪裡欠防護,盲點指認。
3. 風險視角:合規(加州旅遊法、信託、稅務)、平台條款(Meta/微信/蘋果)、版權。
4. 產品視角:一人公司的後台 UX、客人端轉化率、貨架呈現。

規則:
1. 你(外部 AI)看不到程式碼與資料,只有本簡報。回覆時請先列出你的假設,再給意見;不確定就標明不確定,不要編造本專案的細節。
2. 建議請帶取捨與代價,不要只給結論;能量化就量化。
3. 本簡報之外的內部資訊,以 Jeff 現場補充為準。
4. 交流產物是「建議」,不是指令:執行端(Claude 指揮)只把它當輸入之一,一切變更仍走內部驗收與 Jeff 裁決。
5. 不要向 Jeff 索取任何密鑰、帳號、客人個資;他也不會給。

交流方式:Jeff 人肉擺渡。把本簡報貼給外部 AI 開場,之後把具體問題和對方回覆在兩邊往返貼。指揮收到外部意見後會逐點回應(採納/不採納+理由),分歧點交 Jeff 裁決。
